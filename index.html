Testing Types & Methodology

Context for the tags you'll see on every check: how much access you have going in (Black/Grey/White-box), how the analysis is performed (SAST/DAST/IAST), and how each individual test is actually carried out (Manual/Automated). Most real engagements mix all three axes.
By access level

Black-Box Testing

No source code, no credentials, no internal docs. You interact only with the running app, exactly like an external attacker.

Example: the entire Recon category — fingerprinting, directory brute-forcing, and subdomain enumeration all assume zero inside knowledge.

White-Box Testing

Full access to source, architecture diagrams, sometimes design docs. Lets you confirm a vulnerability class exists before ever sending a request.

Example: grepping the codebase for raw SQL string-concatenation or a template-render call fed by user input, ahead of confirming it dynamically.

Grey-Box Testing

Partial knowledge — a low-privilege test account, API docs, or a Swagger file — mixing black- and white-box approaches. Most real client engagements are grey-box.

Example: testing IDOR and BFLA using a real low-priv account the client issued you, without ever seeing the server source.

Manual vs Automated

Manual: a human crafts, sends, and judges each request (Repeater, careful business-logic chaining). Automated: tooling runs at scale (Intruder, ffuf, wpscan, nuclei).

Example: brute-forcing directories is automated (ffuf); confirming a Host header poisoning bug by reading the actual reset email is manual.
By analysis technique

SAST — Static Analysis

Reviewing source code without running it, looking for dangerous sinks, hard-coded secrets, or unsafe patterns.

Example: searching a repo for eval-style functions or unsafe deserialization calls that take user input.

DAST — Dynamic Analysis

Testing the live, running application from the outside, sending real payloads and observing real responses.

Example: sending a UNION-based SQL injection payload to a live search endpoint and reading the returned rows.

IAST — Interactive Analysis

An agent instrumented inside the running app flags issues as you click through it normally, combining static insight with live traffic.

Example: an IAST agent flags a query built by string concatenation the moment you submit a search, even though the HTTP response looked completely normal.
How this maps to the checklist
Recon — almost entirely black-box and automated; it's the phase where you have the least access and lean on tooling.
Burp Suite — a mix of manual crafting (Repeater) and automated tooling (Intruder, extensions); mostly grey-box once you have a session.
Functions — mostly manual, grey-box: you need a real account to walk through the actual business flow.
All Checks — spans every access level, from black-box header tests to white-box-informed deserialization and SSTI probes.
